# tickflow

An open-source **quality-gate and contract-enforcement layer for Kafka-compatible streams**.
Declarative per-topic contracts (Avro schema + semantic rules) are enforced **inline**;
violations are routed to a quarantine topic; a downstream consumer is protected with a
measurable SLO. Every quality claim below is measured by **seeded fault-injection replay**,
reported with 95% confidence intervals, and regenerated by a command you can run.

```
coinbase-ingester ─┐                        ┌─→ trades.valid ──→ barbuilder → DuckDB (+SLO)
                   ├─→ trades.raw ─→ GATE ──┤
kraken-ingester ───┘                        └─→ trades.quarantine (rule_id, detail, offset, …)
```

> **Status: v0.9.** The gate, the SLO experiment, the graded metrics, the CIs, and the telemetry
> dashboard all ship. Quarantine inspection / replay CLI and the live soak do not — see
> [What v0.9 does not ship](#what-v09-does-not-ship). This is a stated v0.9 with a roadmap rather
> than a rushed v0.1.0, per the frozen design's slip valve
> ([ADR-003](docs/decisions.md#adr-003--the-11-slip-valve-is-pulled-quarantinereplay-cli-cut-release-is-v09)).

## Every number here is regenerated, not typed

No figure in this README was written by hand. Each one comes from
[`site/telemetry.json`](site/telemetry.json), which is produced by:

```bash
uv run tickflow export      # → site/telemetry.json + site/index.html
```

The JSON path for each table is given beneath it. The artifact carries a provenance stamp
(commit SHA, fixture content sha256 + parquet sha256, runner, profile, timestamp), so any number
can be traced to the exact fixture and code that produced it. If a number here and the JSON
disagree, the JSON is right and this file is stale.

**All results below are fixture-scale**, produced from the committed synthetic fixture:
100,477 frames (100,000 clean + 477 injected duplicates), seed 42, LRU 10,000, out-of-order
tolerance 5,000 ms. They are not live-traffic numbers. See [Limitations](#limitations).

## Do the gates earn their keep?

The signature experiment (`tickflow slo`): replay the same fault-injected fixture through the
same bar builder twice, once with the gate in front of it and once without.

| Arm | Bars | Violated bars | SLO |
|---|---:|---:|---|
| Gates **ON** | 15,061 | **0** | ok |
| Gates **OFF** | 15,061 | **1,076** | violated |

Which invariants break with the gate removed:

| SLO invariant | Violations |
|---|---:|
| `no_quarantinable` | 1,076 |
| `price_positive` | 144 |

The per-invariant counts sum to 1,220 across 1,076 distinct bars because a single bar can trip
more than one invariant. Violated **bars** is the headline; the invariant counts are the
diagnosis. The two are consistent, not contradictory.

Gates-ON bars are **bit-identical** to the manifest's ground-truth valid projection on the
catchable fault subset — byte for byte, checked by digest — with the 76 designed-miss duplicates
excluded and reported rather than hidden. Why the reference is the valid projection and not the
raw clean fixture is recorded in
[ADR-002](docs/decisions.md#adr-002--the-4-bit-identity-reference-is-the-ground-truth-valid-projection-not-the-raw-clean-fixture).

*Source: `slo` in `site/telemetry.json`. Regenerate: `uv run tickflow slo`.*

## Detection, measured

Graded against the injection manifest's ground truth, every proportion with a 95% bootstrap CI
(B = 10,000, seed 42, percentile):

| Fault class | Rule | n faults | Recall (95% CI) | Precision (95% CI) | Designed misses |
|---|---|---:|---|---|---:|
| `malformed` | R1 | 400 | 1.0000 [1.0000, 1.0000] | 1.0000 [1.0000, 1.0000] | 0 |
| `out-of-range` | R2 | 400 | 1.0000 [1.0000, 1.0000] | 1.0000 [1.0000, 1.0000] | 0 |
| `duplicate` | R3 | 477 | **0.8407 [0.8071, 0.8721]** | 1.0000 [1.0000, 1.0000] | 76 |
| `out-of-order` | R4 | 400 | 1.0000 [1.0000, 1.0000] | 1.0000 [1.0000, 1.0000] | 0 |

**Read the perfect rows skeptically.** The rules are deterministic, so recall on non-boundary
faults is ~100% *by construction*. Those three rows confirm the harness works; they are not an
achievement and are not presented as one.

**The informative row is R3.** Duplicates re-delivered after their key has been evicted from the
dedup window are uncatchable by a finite-memory deduplicator — that is a property of the design,
not a bug — and the injector deliberately plants 76 of them, including a pair straddling the exact
window edge (one key still resident, one just evicted). They drag R3 recall to 0.8407. The number
is lower because the measurement is honest.

*Source: `grade.per_class` in `site/telemetry.json`. Regenerate: `uv run tickflow metrics`.*

## False quarantine, over both denominators

| Denominator | n | Rate (95% CI) | What it is |
|---|---:|---|---|
| All controls | 98,800 | 0.0000 [0.0000, 0.0000] | boundary controls + every untouched clean message |
| **Near-boundary controls** | **460** | 0.0000 [0.0000, 0.0000] | the hard subset |

Reporting only the pooled rate would flatter the result. 98,340 of those 98,800 are ordinary
clean messages no rule comes near. The 460 near-boundary controls are the ones engineered to sit
one representable step from a quarantine decision — prices exactly on the inclusive R2 bound,
skews of 4.9 s and exactly 5.0 s against a 5.0 s tolerance, and the dedup-window edge — and they
are where an off-by-one in a comparison operator would actually surface. So both are published,
each with its own n and CI.

**The `[0.0000, 0.0000]` intervals are a method artifact, not a proof.** A percentile bootstrap
over a sample containing zero events resamples zeros forever. It cannot see events it never
observed. Read them as "no false quarantine in n trials"; the honest one-sided ceiling on an
all-zero sample is the rule of three, ≈3/n — about **0.0065** on the 460 near-boundary controls
versus **0.00003** over all 98,800. That ~200× gap is exactly why the two are not pooled.

*Source: `grade.false_quarantine_rate` in `site/telemetry.json`.*

## Completeness and throughput

| Measure | Value |
|---|---:|
| Frames accounted for | 100,477 |
| Routed valid | 98,876 |
| Quarantined | 1,601 |
| Loss / duplicate delivery | 0 / 0 |
| Gate throughput | ~2.2 × 10⁵ msg/s |

Every input frame is accounted for exactly once across valid + quarantine.

Throughput is quoted to one significant figure on purpose: it is wall-clock timing on a shared
laptop and drifts a few percent run to run (observed 214k–225k msg/s across four consecutive
runs), so more digits would imply a precision the measurement does not have. The exact figure for
any given run is in the artifact. It is **in-process rule-engine throughput only** — no broker,
network, or fsync cost — measured on Apple silicon (Darwin arm64, Python 3.12). It answers "is
the rules engine fast enough that gating is not the bottleneck", and it is **not** a system
capacity figure and not independently reproducible.

*Source: `grade.completeness` and `throughput` in `site/telemetry.json`.*

## Dashboard

[`site/index.html`](site/index.html) is a static, dependency-free page (no JavaScript, no
external assets) rendered from the same telemetry JSON and deployed to GitHub Pages by
[`.github/workflows/pages.yml`](.github/workflows/pages.yml).

It shows **pipeline telemetry only** and states "last refreshed &lt;timestamp&gt;" — never a
refresh cadence, because the Actions cron is best-effort and GitHub does not guarantee scheduled
runs fire on time or at all. CI fails the build if a cadence claim ever appears on the page.

## Why this exists

The streaming *pipeline* is a solved problem; the *enforced quality gate* in front of it is
not. Inline stream-quality rules sit behind enterprise tiers (Confluent's docs state schema
rules are "only available on Confluent Enterprise, not on the Community edition"), and SQL
engines like Flink SQL and ksqlDB don't execute such rules at all. Teams that need field-level
semantic enforcement have built it in-house (Grab, Nov 2025: field-level semantic rules across
100+ Kafka topics with violations routed to a dedicated topic). Data contracts, meanwhile, have
largely remained *descriptive artifacts rather than interfaces with failure semantics* (Data
Engineering Weekly, Jan 2026). tickflow is that missing gate, built as a contract with failure
semantics.

**Market data is the demo domain, not the product.** It is free, 24/7, and naturally hostile —
duplicates, gaps, out-of-order delivery, and cross-venue divergence. The product is the gate.
If the gates were deleted, the remaining pipeline would be worthless by design.

**Data-handling boundary (a ToS requirement, not a style choice):** the exchanges' market-data
terms prohibit redistributing the data or derived works. Therefore **no captured market data is
ever committed to this repo**, and **the public dashboard shows pipeline telemetry only** —
counts, rates, and verdicts, never prices, bars, volumes, or anything derived from market-data
values. `tickflow export` enforces this mechanically: it walks the entire payload and raises
rather than writing a market-data field, and CI re-checks the built artifact before deploying.

## The contract

One contract, six declarative rules ([contracts/](contracts/)) — every threshold, bound, and
disposition read from YAML, nothing hard-coded in the engine:

| ID | Rule | On violation |
|----|------|--------------|
| R1 | schema (Avro decode) | quarantine `malformed` |
| R2 | range (per-symbol static bounds; price/size > 0) | quarantine `out-of-range` |
| R3 | duplicate (LRU window per stream) | quarantine `duplicate` |
| R4 | out-of-order (event-time watermark tolerance) | quarantine `out-of-order` |
| R5 | gap (watermark jump on an active stream) | **alert only** |
| R6 | divergence (cross-venue mid deviation) | **alert only** |

R5 and R6 are structurally incapable of quarantining — the config loader rejects any YAML that
tries. A genuine market move diverges across venues before it converges, and a missing print is
not a bad message; quarantining for either would punish true data for reality's behavior.
Detection is not rejection, and the gate distinguishes them.

Rules evaluate against the per-stream **event-time watermark**, never wall clock — so the same
messages in produce the same verdicts out, live or in replay. That determinism is what makes
every measurement above meaningful.

The Avro contract is registered with BACKWARD compatibility and checked in CI. Verified locally
against a real Redpanda Schema Registry (v24.2.7), including that `tickflow contract check`
correctly **fails** with exit 1 on a BACKWARD-breaking change — a compatibility gate that only
ever answers "compatible" is indistinguishable from a no-op.

## Limitations

Stated plainly, because a measured claim is only worth what its caveats admit.

- **No latency claim is made anywhere.** The local Redpanda `dev` profile bypasses fsync — per
  Redpanda's own documentation, which notes this yields "unrealistically fast clusters" — so any
  latency measured against it would be meaningless. tickflow therefore publishes no latency
  numbers at all rather than publishing flattering ones with a footnote.
- **The committed fixtures are synthetic.** They are generated by a seeded, integer-driven,
  cross-platform-deterministic generator and pinned by content digest. **No captured real market
  data is committed to this repository, ever** (exchange ToS). Synthetic data means the fault
  distribution is the one the injector chose, not the one production would hand you: real feeds
  fail in ways this fixture does not contain.
- **R3 recall is below 100% by construction.** The injector deliberately plants duplicates that
  arrive after their key has been evicted from the finite dedup window. No finite-memory
  deduplicator can catch them. They are counted as misses rather than excluded from the
  denominator, which is why R3 reads 0.8407 instead of 1.0000.
- **The SLO comparison numbers are fixture-scale, not live-traffic.** 0 vs 1,076 violated bars is
  what happens when *this* fixture, with *this* injected fault mix at ~2%, meets *this* rule set.
  It demonstrates that the gate prevents corruption that would otherwise reach the bars. It does
  not predict how many bars a live feed would corrupt.
- **Perfect recall on R1/R2/R4 is a property of deterministic rules**, not evidence of a
  sophisticated detector. The rules are exact predicates; the faults are drawn to violate them.
- **The zero false-quarantine CIs are degenerate.** See the rule-of-three ceiling above.
- **Single-node, at-least-once, `double` not decimal.** One broker, no cluster failure testing.
  Delivery is at-least-once, so exactly-once accounting under kill/restart is asserted only
  in-process, not against a real broker (that proof is roadmap, not shipped). Prices are IEEE
  `double`; a production financial system should use a decimal type.
- **Throughput is in-process only** and excludes every cost a real deployment has.
- **The dashboard refresh is best-effort.** Scheduled GitHub Actions runs are not guaranteed.

## What v0.9 does not ship

Cut via the frozen design's slip valve so that the measurement work above could be done properly
([ADR-003](docs/decisions.md#adr-003--the-11-slip-valve-is-pulled-quarantinereplay-cli-cut-release-is-v09)):

- `tickflow quarantine` ls/show/stats and `tickflow replay --fixture` — inspection convenience
  over envelopes the gate already emits and faults the metrics already grade.
- The kill/restart-mid-replay exactly-once completeness proof against a real broker.
- The end-to-end integration lane in CI (still `if: false`).
- The live soak. **No live-soak numbers are published anywhere.**

The gate, the SLO experiment, and the CIs — the parts the slip valve may never touch — all ship.

## Reproduce it

```bash
uv sync                                  # install (Python 3.12)
uv run tickflow fixture verify           # check the committed fixture against its pins
uv run tickflow slo                      # the gates-ON/OFF experiment
uv run tickflow metrics                  # graded detection + false quarantine, with CIs
uv run tickflow export                   # regenerate site/telemetry.json + site/index.html
```

Development:

```bash
uv run ruff check .     # lint
uv run mypy             # type-check (strict, on src/)
uv run pytest           # tests + coverage
docker compose --profile dev up -d    # single-node Redpanda + Schema Registry
```

`dev` and `bench` compose profiles are documented in
[docs/architecture.md](docs/architecture.md) §3. The `dev` profile bypasses fsync and its timings
are never quoted. **If a broker-derived number is ever published, it comes from the `bench`
profile only** — that rule stands for the roadmap work (integration lane, soak). It is vacuous
today: no published number in v0.9 depends on a broker at all.
